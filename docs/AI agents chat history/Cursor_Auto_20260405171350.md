## How your model differs today (important constraint)

- **`Lexeme.type`** (`LEXEME` | `INFLECTED_FORM` | `FORM_OF`) is decided by **`guessLexemeTypeFromTemplates()`** from **templates**, not from the PoS heading (`wiktionary-core.ts`). Form-of templates like **`abbreviation of`** are already in the form-of family (`form-of-predicates.ts`).

So: **you cannot honestly set `type` to ŌĆ£non-lemmaŌĆØ from a section title alone** without either (a) explicit wikitext templates, or (b) relaxing the ŌĆ£no guessingŌĆØ rule. The robust approach is to **enrich classification fields**, not to fake `Lexeme.type` from headings.

## Design direction: split ŌĆ£grammar PoSŌĆØ vs ŌĆ£lexicographic section classŌĆØ

## 2. Where they **align**

1. **En.wiktionary focus** (scraper README: English Wiktionary only; SDK is built around en.wiktionary with language sections inside the page).
2. **Rich section model** aligned with [Wiktionary:Entry layout](https://en.wiktionary.org/wiki/Wiktionary:Entry_layout): etymology, pronunciation, definitions, usage notes, inflection, relations, translations, etc. ScraperŌĆÖs `types.ts` documents this explicitly; your spec does the same in prose.

| **Testing / tooling** | Mocha + smaller surface. | Vitest, golden tests, decoder coverage, CLI, playground, introspection scripts. |

## 4. Functions, capabilities, ease of use

- **Ease of use (simple case):** **Scraper wins** for ŌĆ£give me something for this wordŌĆØ in a few lines: `get('word')` + **Cheerio** only.

## 5. Maintainability and manageability

- **Scraper:** Small codebase, **TOC-driven** parsing. **Fragile** if MediaWiki changes HTML structure, TOC markup, or class names; selectors live in `constants/selectors.ts`. Easier to read end-to-end, harder to guarantee across skin/CSS/Parsoid changes.
- **SDK:** **Heavier** but **more stable** at the data layer because it keys off **wikitext** and template names. Maintenance is **registry growth**, schema parity, and API compatibilityŌĆönot DOM breakage.
