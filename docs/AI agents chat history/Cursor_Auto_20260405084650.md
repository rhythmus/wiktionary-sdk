Here’s a concrete follow-up menu you could take in slices, all aimed at **smaller files and clearer ownership** while **keeping one normative registration sequence** (still enforced by `registry-decoder-order.test.ts`).

## 1. **Orchestrator-only `register-all-decoders.ts`**

Keep a **thin** `registerAllDecoders(reg)` that does nothing but call family functions **in the exact historical order**:

```text
registerCoreAndPronunciation(reg);      // store-raw, ipa, hyphenation, alt-forms-section
registerHeadwordsElNlDe(reg);           // el… through de-verb-head
registerFormOfAndWikidata(reg);
registerTranslations(reg);
registerSensesPipeline(reg);            // senses + everything that must stay after shared sense helpers…
// …etc.
```

Each family module **only** receives `reg` and calls `reg.register({…})` in the right sub-order. **No** reordering inside or across those calls relative to today.

That gives you a **readable map** of the registry without changing behaviour.

---

## 2. **Slice by “family” (suggested boundaries)**

These line up with how the file is already commented and how dependencies cluster:

| Slice | Rough contents | Notes |
|--------|----------------|--------|
| **Core + pronunciation** | `store-raw-templates`, `ipa`, `hyphenation`, `alternative-forms-section` | Hyphenation helpers stay here or in `registry/hyphenation.ts` |
| **Headwords (el / nl / de)** | All `*-head` through `de-verb-head` | `GENDER_MAP` used by nl/de (and later el-noun-gender, la-noun): define **once** in `registry/gender-map.ts` or keep in the **last** headword file that needs it before first use |
| **Form-of + Wikidata** | `form-of`, `wikidata-p31` | `TAG_LABEL_MAP` / `buildFormOfDisplayLabel` → `registry/form-of-label.ts` or stay next to form-of |
| **Translations** | `translations` + `parseTranslationsFromBlock` |
| **Senses** | `senses` + **all** helpers only used by sense parsing (`parseSenses`, `parseLbTemplate`, `formatUsageNoteLine`, …) | Biggest blob; highest payoff |
| **Greek morphology + Latin head** | `el-verb-morphology`, `el-noun-gender`, `la-noun-head` + `decodeTransitivity`, `VERB_PART_PARAMS`, `decodeLaNounGenderFromTemplate` |

You can merge or split further (e.g. “sections” vs “pronunciation”) depending on file size targets.

## 3. **Shared helpers (avoid duplication and cycles)**

Several helpers are used **across** families. Options:

- **`registry/section-extract.ts`** — `extractSectionByLevelHeaders`, `parseSectionLinkTemplates`, maybe `matchesSectionHeading` if you want one “section” home.
- **`registry/gender-map.ts`** — `GENDER_MAP` + small normalisers if you want nl/de/el/la to import the same map.
- **`registry/form-of-display-label.ts`** — `TAG_LABEL_MAP`, `tagsToLabel`, `defaultFormOfKindLabel`, `buildFormOfDisplayLabel` (used by form-of decoder and sense-line gloss logic).

Rule of thumb: **pure functions + constants** → small `*.ts` modules; **`reg.register`** → family `register-*.ts` files.

---

## 4. **What else you could do (beyond file splits)**

- **`export { registerAllDecoders }` from `registry.ts`** — so tests or tools can build a **second** `DecoderRegistry` (isolated tests) without importing an internal path (optional API surface).
- **Doc-only “registration map”** — generate `docs/registry-inventory.md` decoder table from `EXPECTED_DECODER_IDS` or from a single `DECODER_IDS` const **imported by the test** (one source of truth).
- **Stricter CI** — a script that diffs `registry.getDecoders().map(d => d.id)` against committed JSON (same idea as the test, but for release checks).
- **Later Phase 4.4 tie-in** — as you shrink `DECODER_EVIDENCE_ALLOWLIST`, family files make it obvious **which fixture** belongs to which decoder group.

---

## 5. **Order of work (low risk → higher)**

1. Extract **pure** helpers (`section-extract`, `gender-map`, `form-of-display-label`) **without** moving any `reg.register` — behaviour unchanged, diffs easy to review.  
2. Replace `register-all-decoders.ts` body with **sequential calls** to new `registerX(reg)` in **one PR per family** (or 2–3 families per PR), running tests + order test each time.  
3. Optionally shrink `register-all-decoders.ts` to **~40–80 lines** of orchestration only.

---

## 6. **What I would *not* do in the same pass**

- Reordering registrations “to group by family” if that changes the **global** order (breaks merge semantics).  
- Splitting **one decoder’s** `decode` across files without a strong reason (harder to grep).  
- Introducing a **runtime plugin** registry unless you have a product need (adds API and testing surface).
