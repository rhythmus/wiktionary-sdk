## What was going wrong

Several tests use **`vi.mock("../src/index", …)`** and only replace the exported **`wiktionary`** with `vi.fn()`. Anything imported from `../src/index` in the test file then sees the mock.

But **`library.ts`**, **`stem.ts`**, and **`morphology.ts`** do `import { wiktionary } from "./index"`. With the circular graph created by `importOriginal()`, that binding often stays tied to the **real** `wiktionary`, which still calls **`fetchWikitextEnWiktionary`** → **`mwFetchJson`** → real **`fetch()`** to `en.wiktionary.org`, plus the **100 ms rate limiter** between requests.

So you got:

1. **Very slow tests** (hundreds of ms per call, multiple throttled requests for paths like `lemma` → `wiktionary` → `lemma` → `wiktionary`).
2. **Misleading speed on later tests**: the first live fetch filled **`MemoryCache`**, so later tests in the same run could look “fast” even though they were still using the network stack indirectly.
3. **Apparent hangs**: on a bad or blocked network, **`fetch()`** can stall until Vitest’s **`testTimeout` (30 s)**.

Concrete signal: in isolation, **`should fetch interwiki langlinks`** spent **~240 ms** (live API + throttle) while the next tests in the same file were **~2 ms** (cache hits).

## What we changed

**`test/enrichment.test.ts`**, **`test/auto.test.ts`**, and **`test/stem.test.ts`** now **`vi.mock("../src/api", …)`** and stub **`fetchWikitextEnWiktionary`** / **`fetchWikidataEntity`** with **`beforeEach`** implementations that return **fixtures or small inline wikitext**. Real `wiktionary` can still run for `interwiki`, `lemma`, etc., but **no HTTP** and **no cross-test reliance on cache**.

After the change, the same enrichment file shows **`should fetch interwiki langlinks` ~32 ms** (decode work only, no network).

## Note on current failures

**`test/library.test.ts`** still has **two failing tests** (`derivations` missing export, **`derivedTerms`** shape vs `string[]`). Those failures were already present in the run you had; this change does not touch that file.

If you want, we can next either fix those library tests/exports or add the same **`api` mock pattern** to **`library.test.ts`** for extra safety when `lemma` touches the real `wiktionary` binding.

Here’s a concise assessment grounded in how this repo is set up (source-faithful extraction, registry decoders, no linguistic guessing) and what the tests actually exercise.

## Are they robust, correct, and relevant?

**Relevant — yes, in the right places.**  
Parser, registry, fixture-based integration, Phase 2/3 (senses, relations, cache, rate limiter), and schema checks all line up with the architecture in `AGENTS.md` and the “extract what’s in the wikitext” goal.

**Correct — mostly for what they assert.**  
Where tests check concrete behavior (template parsing, fixture round-trips, schema validity), they can catch real regressions. Where they assert against **hand-built `FetchResult` mocks** (many `library.test.ts` cases), they are correct for “given this normalized blob, wrappers return X” — not for “given real en.wiktionary wikitext, we always produce that blob.”

**Robust — mixed.**  
- **Weak spot:** reliance on **partial `index` mocks + `api` stubs** is easy to get wrong (we already saw real `wiktionary` still running for `library`/`interwiki`). The suite is more robust *after* the API-layer stubs, but the pattern is still fragile for future contributors.  
- **Flake risk:** `bench.test.ts` ties behavior to **wall-clock thresholds** (e.g. avg &lt; 10 ms). That’s useful as a smoke signal locally but can fail on loaded CI or different hardware; it’s not a guarantee of correctness, only of rough performance.

## Do they “guarantee correct and true outputs”?

**No — and no test suite fully can for this domain** without either freezing corpora or live golden data. Specifically:

- They do **not** prove that every decoder matches live Wiktionary for all templates in the wild.  
- They do **not** prove semantic truth of glosses (only that strings match fixtures or mocks).  
- Schema tests prove **shape**, not **fidelity** to the source page.

What they **do** guarantee, when green:

- Parsed structures and registry behavior match **specified fixtures and examples**.  
- Many **invariants** (brace-aware parsing, template decoding paths you’ve written tests for, cache/rate-limiter behavior, etc.).  
- **README examples** stay consistent with **fixture-backed** flows in `readme_examples.test.ts`.

So the honest claim is: **regression safety on encoded expectations**, not **omniscient correctness** against the live wiki.

## What I’d propose to improve the suite

1. **Prefer one mocking strategy per layer**  
   - Either: **mock only `api`** (fixtures → real `wiktionary` → assert `FetchResult`), or **mock only `wiktionary`** but then **don’t** test paths that go through `library`’s internal binding without the same mock.  
   Document that in `AGENTS.md` or a short `test/README.md` so the partial-index pattern doesn’t creep back.

2. **Golden / snapshot tests for normalized entries**  
   For a **small, versioned set** of fixture titles (or frozen wikitext files), snapshot the **stable subset** of `NormalizedEntry` / YAML (e.g. ids stripped or normalized) to catch unintended decoder or merge changes.

3. **Decoder coverage matrix**  
   Extend something like `template-introspect` (or a test-only harness) to assert: **every registered decoder** has at least one **fixture or unit test** with expected fields. Gaps become explicit.

4. **Separate “fast unit” vs “slow integration”**  
   - `vitest run test/unit/**` — no I/O, tight timeouts.  
   - Optional `test/integration/**` or `test/fixtures/**` with slightly higher timeout.  
   Keeps CI predictable and makes “guarantees” clearer per tier.

5. **Soften or relocate perf tests**  
   Replace fixed ms thresholds with **relative regression** (vs baseline in CI artifact) or mark `bench.test.ts` as **non-blocking** / separate `npm run test:perf` so correctness CI doesn’t depend on CPU load.

6. **Property-style or invariant tests for the parser**  
   e.g. round-trip: “parsed template spans are inside source length,” “no overlapping template ranges,” “re-parsing extracted template raw equals input slice.” Cheap and high value for the custom parser.

7. **`derivations` vs spec**  
   If the spec promises `derivations()`, add **`export const derivations = derivedTerms`** (or deprecate the spec term) so **API, tests, and docs** agree — avoids drift and mistaken “missing API” reports.

8. **Optional: recorded HTTP (VCR-style) for rare live checks**  
   A **single** optional job that replays recorded API JSON against `fetchWikitext` parsing — not default CI — to detect MediaWiki format changes without flaky live runs.

---

**Bottom line:** The suite is **on-mission** and **good for regression** on fixtures and core parsing/registry behavior; it does **not** by itself guarantee globally correct or “true” dictionary output. The biggest wins are **clearer layering of mocks**, **golden tests on fixtures**, **explicit decoder coverage**, and **decoupling perf gates from correctness CI**.
