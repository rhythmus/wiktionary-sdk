## 1. Fix real mismatches inside `normalized-entry.schema.json` (bugs, not “nice to have”)

### 1.1 `$defs.RichEntry` does not match `RichEntry` in `types.ts`

```486:508:src/types.ts
export interface RichEntry {
  headword: string;
  pos: string;
  type?: LexemeType;
  form_of?: Lexeme["form_of"];
  morphology?: any;
  headword_morphology?: Lexeme["headword_morphology"];
  pronunciation?: Pronunciation;
  hyphenation?: Hyphenation;
  etymology?: EtymologyData;
  senses?: Sense[];
  relations?: SemanticRelations;
  derived_terms?: SectionWithLinks;
  related_terms?: SectionWithLinks;
  descendants?: SectionWithLinks;
  usage_notes?: string[];
  references?: string[];
  inflection_table?: InflectionTable;
  translations?: Lexeme["translations"];
  wikidata?: WikidataEnrichment;
  images?: string[];
  source: WiktionarySource;
}
```

In the JSON Schema, `RichEntry.etymology` is currently an **array of objects**, but the implementation uses **`EtymologyData`** (`chain` / `cognates` / `raw_text`) — same as `Lexeme`. That `$ref` should be `#/$defs/EtymologyData`, not an array.

**Add (do not remove existing RichEntry keys):**

- `headword_morphology` → `$ref: "#/$defs/HeadwordMorphology"` (mirrors `library.richEntry`, which copies `lexeme.headword_morphology`).
- `images` → `array` of `string` (same as `Lexeme.images`).

Keeping `morphology` as a loose object is fine (`richEntry` fills a small `{ aspect, voice }` object).

### 1.2 `$defs.AlternativeForm` is narrower than `Lexeme.alternative_forms[]` in `types.ts`

Types include optional `type` and `labels`:

```417:425:src/types.ts
  alternative_forms?: Array<{ 
    term: string; 
    qualifier?: string; 
    raw: string;
    /** The specific type of alternative form (e.g. "polytonic", "archaic"). */
    type?: string;
    /** Labels associated with the variant (e.g. ["rare", "poetic"]). */
    labels?: string[];
  }>;
```

**Extend** `AlternativeForm` with optional `type` (`string`) and `labels` (`array` of `string`). Retain `term`, `qualifier`, `raw` as today.

### 1.3 `FetchResult.metadata` and top-level `debug` are under-specified vs implementation

`wiktionary-core` sets:

```525:534:src/wiktionary-core.ts
    const out: FetchResult = {
        schema_version: SCHEMA_VERSION,
        rawLanguageBlock: lang === "Auto" ? qPage.wikitext : searchSections[0]?.block || "",
        lexemes: merged,
        notes: resolvedPageTitleNote && debugDecoders ? [resolvedPageTitleNote] : [],
        metadata: {
            categories: (qPage as any).categories || [],
            langlinks: (qPage as any).langlinks || [],
            info: (qPage as any).info || {},
        },
    };
```

**Tighten without removing:**

- `metadata.langlinks.items` should be the same object shape as `Lexeme.langlinks` items (`lang` + `title`, `additionalProperties: false`), not `{}`.
- `metadata.info` can reuse the same property set as `Lexeme.metadata` (`last_modified`, `length`, `pageid`, `lastrevid`) plus `additionalProperties: true` if you want to allow future MediaWiki fields (additive).

**`debug`:** introduce `$defs.DecoderDebugEvent` matching `types.ts`:

- `decoderId`, `matchedTemplates` as array of `{ name, raw }`, `fieldsProduced` as `string[]`.

Nested array shape matches `debug?: DecoderDebugEvent[][]`.

### 1.4 `schema_version` on `FetchResult`

The engine always sets `schema_version`. Tests already validate with it. **Add** `schema_version` to the root `required` array **in addition to** existing required fields (do not remove `rawLanguageBlock`, `lexemes`, `notes`).

### 1.5 `TranslationItem.params`

It is `{}` today while `types.ts` uses `params: any`. For an exhaustive but honest schema, **replace** the empty schema with something like:

- `oneOf`: strict `TemplateParams` **or** `additionalProperties: true` object,

so validators accept both normalized template params and oddball cases, without dropping the field.

## 2. Additive extensions for a more complete lexicographic domain model

These are not necessarily wrong today; they make the canonical schema document the full surface the codebase actually uses or implies.

### 2.1 `EtymologyData`: optional legacy `links`

`library.extractEtymologySteps` reads `(lexeme.etymology as any).chain || (lexeme.etymology as any).links`. **Add** optional `links` with the same item type as `chain` (or `$ref` to `EtymologyLink[]`), marked in `description` as deprecated / compatibility. Keep `chain` as the primary field.

### 2.2 Document `Lexeme.id` (and optionally constrain)

Today it is a free `string`. You could **add** a `description` and an optional `pattern` reflecting the conventional `{lang}:{title}#E{n}#…` form used in fixtures (only if you are willing to keep the pattern stable).

### 2.3 `lexicographic_section` vocabulary

It is a free `string` in schema; the closed set is implied by `src/lexicographic-headings.ts` (many slugs: `verb`, `suffix`, `han_character`, …). For exhaustiveness **without** breaking open-world data:

- Add a `$defs.LexicographicSectionSlug` **enum** listing every `section_slug` registered in `lexicographic-headings.ts`, and set `Lexeme.lexicographic_section` to `oneOf`: that enum **or** `string` (so unknown future headings still validate). That adds documentation value without removing the string branch.

### 2.4 `form_of.tags` and inflection dimensions

Tags are opaque strings in output. Optionally **add** parallel optional fields (all optional, populated only when decoders fill them) such as `grammar_traits` matching `GrammarTraits` from `morphology.ts` — **only if** you add the same to `types.ts` and the pipeline. If you do not want new runtime fields yet, limit the schema change to a **`description`** on `tags` referencing `parseMorphologyTags` / form-of abbreviations.

### 2.5 `principal_parts` slot names

Already `additionalProperties: string` on `HeadwordMorphology.principal_parts`. **Enrich descriptions** with examples from decoders (e.g. Greek `present`, `aorist_active`, `stem` inside `principal_parts` from `register-inflection-stems.ts`). No removal.

### 2.6 Top-level `$defs` for convenience API types (optional second schema or same file)

Public API parity (`AGENTS.md`) includes `library.ts` shapes not in `FetchResult`:

- `LexemeResult<T>` (for documentation, use `value: {}` or generic-less JSON Schema).
- `GroupedLexemeResults` as array + `order` + `lexemes` map.

Morphology API:

- `ConjugateCriteria`, `DeclineCriteria`, `GrammarTraits` as small `$defs` (input side, not lexeme output).

Adding these as **`$defs` only** (not required on `FetchResult`) keeps the main output schema stable while making the “lexicographic domain” documentation exhaustive.

### 2.7 `WiktionarySource.last_modified` format

Schema uses `format: "date-time"`. MediaWiki often returns timestamps that are not always strict RFC 3339. **Option A:** relax to `string` + description. **Option B:** keep `date-time` and document that consumers should normalize. Prefer additive: add `examples` and a note rather than removing `format`.

Would we loose anything if we were to keep and maintain our all-encompassing, canonical and authoritative lexicographic domain model schema (as our single source of truth) in YAML instead of JSON?

---

**Cursor**


You would not lose anything **about the lexicographic domain model itself** if that model were authored and maintained in YAML: the *concepts* (types, relations, constraints) are independent of whether you write them in JSON or YAML.

What you **would** lose or complicate is mostly **tooling and process**, unless you treat YAML as **source** and **always** emit a **JSON Schema** artifact for the rest of the world.

### If YAML were the *only* artifact you ship

- **Validator ecosystem**: Mainstream JSON Schema validators (e.g. Ajv) expect JSON (or an object tree equivalent to parsed JSON). You’d depend on YAML parsing first, then validation—extra step, another parser surface, and YAML’s edge cases (duplicate keys, `yes`/`no`, `1.10` vs `1.1`, anchors).
- **Interoperability**: Many consumers, docs sites, and linters assume a `.schema.json` URL or file; you’d be non-standard unless you publish generated JSON anyway.
- **Determinism**: JSON is minimal and unambiguous; YAML authoring mistakes are easier and sometimes silent.
- **Spec framing**: JSON Schema is **specified** as a JSON document. YAML is fine as a *syntax* for that same structure, but “the” standard interchange form is still JSON-shaped trees.

### If YAML were the **canonical source** and JSON were **generated** (recommended pattern if you move)

- You **don’t** lose model fidelity: same `$defs`, `required`, `oneOf`, etc., after transpilation.
- You **keep** Ajv, CI, and `npm test` schema checks unchanged by **committing** or **building** `normalized-entry.schema.json` from YAML.
- You **gain** nicer authoring (comments, readability) **if** your team agrees on one YAML style and a single pipeline (e.g. `yaml → json` in CI + pre-commit).

### Summary

- **Domain model**: no inherent loss from YAML vs JSON.
- **Authority**: you can make YAML the *human* source of truth only if **JSON Schema JSON** remains the **machine** contract you validate against and publish—or you accept weaker tooling and more integration friction.

For this repo, the low-risk stance is: **YAML (or split modular YAML) as author-time source + generated `schema/normalized-entry.schema.json` as the authoritative artifact for validators and consumers**, with a test that regeneration is clean.

## 1. Organize by **pipeline stage** (what the codebase actually does)

A structure that matches the mental model “fetch → parse → decode → merge → present → convenience API” tends to stay navigable as the project grows:

| Area | Role | Today (roughly) |
|------|------|-------------------|
| **Ingress** | HTTP, cache, rate limits, normalization of API payloads | `api.ts`, `cache.ts`, `rate-limiter.ts`, `server-fetch.ts` |
| **Parse** | Wiktext sections, templates, language/heading mapping | `parser.ts`, `lexicographic-headings.ts` |
| **Decode** | Template → structured patches | `registry.ts` + `registry/*` |
| **Orchestrate** | Section walk, merge, lemma resolution, enrich | `wiktionary-core.ts`, `form-of-parse-enrich.ts` |
| **Model** | Types and constants tied to the output contract | `types.ts`, `constants.ts` (and schema parity) |
| **Present** | Handlebars, grouping, form-of display strings | `formatter.ts`, `lexeme-display-groups.ts`, `form-of-display.ts`, `templates/*` |
| **Convenience** | Thin wrappers over `wiktionary()` | `library.ts`, `morphology.ts`, `stem.ts`, `wrapper-invoke.ts` |
| **Infra** | Shared pure helpers | `utils.ts` |

A possible **target layout** (names illustrative):

```text
src/
  index.ts                    # public barrel only
  model/                      # types + SCHEMA_VERSION + small type guards
  ingress/                    # api, cache, rate-limiter, server-fetch
  parse/                      # parser, lexicographic-headings
  decode/                     # registry.ts + registry/* (move current registry/)
  pipeline/                   # wiktionary-core, form-of-parse-enrich
  present/                    # formatter, form-of-display, lexeme-display-groups, templates/
  convenience/                # library (split), morphology, stem, wrapper-invoke
  infra/                      # utils, constants (or keep constants next to model)
```

Nothing here requires changing the **published** API if `index.ts` keeps re-exporting the same symbols; you mostly move files and fix imports (plus tests/CLI/webapp that import `../src/...`).

---

## 2. Biggest maintainability wins: **split the large files**

Line counts (approximate) point to where structure pays off first:

- **`library.ts` (~837 lines)** — Split by **feature family** (e.g. `library/translate.ts`, `library/relations.ts`, `library/etymology.ts`, `library/rich-entry.ts`, `library/grouped-results.ts`) with a thin `library/index.ts` that re-exports what `index.ts` exposes today. That makes parity with CLI/webapp (`AGENTS.md`) easier to audit file-by-file.
- **`formatter.ts` (~1020 lines)** — Split into **registration/helpers** vs **per-mode render** (e.g. `present/format-html.ts`, `present/format-md.ts`, `present/handlebars-helpers.ts`) and keep a small `formatter.ts` facade.
- **`types.ts` (~598 lines)** — Split into **domain slices** (`model/lexeme.ts`, `model/fetch-result.ts`, `model/sense.ts`, `model/wikidata.ts`, `model/decoders.ts`, …) and a **`model/index.ts`** barrel so external imports stay `from "./types"` via a one-line `types.ts` re-export during migration.
- **`wiktionary-core.ts` (~562 lines)** — Extract **named phases** into `pipeline/extract-language-block.ts`, `pipeline/build-lexeme-base.ts`, `pipeline/merge-lemma-resolution.ts`, etc., even if the main file stays the single entry for `wiktionary()` / `wiktionaryRecursive()`.

`parser.ts` (~377) and `morphology.ts` (~370) are candidates for a **second** wave of splits (e.g. template parsing vs section splitting).

---

## 3. Keep what already works: **`registry/`**

The `registry/` + `register-*.ts` family is already modular and matches AGENTS. Prefer **moving the folder as a unit** under `decode/registry/` (or keep path `src/registry/` and only document it as the “decode layer”) rather than fragmenting it further unless a single file crosses ~400–500 lines.

---

## 4. Tackle **import cycles** on purpose

You already avoid `index` from `wiktionary-core`; **`morphology.ts` ↔ `library.ts`** is a known tight coupling. A re-org is a good moment to:

- Pull **shared, low-level** pieces (e.g. “call `wiktionary` with these options”) into a tiny **`pipeline/internal-fetch.ts`** or **`convenience/wiktionary-helpers.ts`** imported by both, **or**
- Restructure so **`morphology` does not import `library`** (only `wiktionary-core` + small helpers).

Document allowed dependency direction on a **one-page diagram** in `docs/` or a short `src/ARCHITECTURE.md`: e.g. `convenience → pipeline → decode|parse → ingress`, `present → model`, no `parse → convenience`.

---

## 5. **Migration strategy** (so it stays shippable)

1. **Introduce folders and move files in small PRs** (e.g. only `ingress/` first), updating imports; keep **`src/index.ts` exports identical**.
2. **Optional compatibility shims**: leave `src/parser.ts` as `export * from "./parse/parser"` for one release if you want to soften churn for forks (usually unnecessary if you control all importers).
3. **Split `library.ts` and `types.ts` last or in dedicated PRs** — they touch the widest surface (tests, webapp, CLI).
4. After each phase: **`npm run build` + full `npm test` + quick grep for deep imports** in `webapp/` and `cli/`.

---

## 6. What I would **not** do first

- Renaming **public** symbols or paths in `package.json` `exports` without a major version — keep the external contract on `index.ts`.
- Spreading **decode** logic back into `parser.ts` — AGENTS’ registry rule is worth preserving.
- A big-bang “everything moves” PR — high conflict and review cost.

---

**Summary:** The most **revealing** layout is **stage-based folders** (ingress / parse / decode / pipeline / model / present / convenience) plus **splitting `library.ts`, `formatter.ts`, and `types.ts`**, while keeping **`registry/`** as the decode backbone. **Manageable** rollout means **incremental moves** and **explicit dependency rules** to trim cycles like morphology ↔ library.

If you want this executed in the repo next, a sensible first PR is **`ingress/` + `parse/`** (low coupling, clear boundaries), then **`model/`** from `types.ts`, then **`library/`** slices.
